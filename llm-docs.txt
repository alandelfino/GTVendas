## AUTENTICAÇÃO

A API suporta dois mecanismos de autenticação:

### 1. Sessão (Web)
Login via POST /api/login. O servidor retorna um cookie de sessão que deve ser enviado automaticamente nas requisições subsequentes.

### 2. JWT Bearer Token (Mobile / Apps Nativos)
Login via POST /api/mobile/login. Retorna accessToken (15 min) e refreshToken (30 dias).
Envie em todas as requisições: Authorization: Bearer <accessToken>
Quando o accessToken expirar (erro 401), use POST /api/mobile/refresh para renovar.

### 3. API Token (Integrações / Webservice)
Token permanente gerado pelo usuário master em /api/tokens.
Envie: Authorization: Bearer <apiToken>
Usado exclusivamente nos endpoints de escrita do ERP (POST/PATCH/DELETE em /api/erp/*).

### Níveis de acesso:
- master: acesso total a todos os endpoints
- webservice: acesso a leitura do ERP e endpoints tokenAuth
- user: acesso às rotas /api/rep/* (vinculado a um representanteId)

## MÓDULO: AUTENTICAÇÃO WEB

### POST /api/login
Autentica o usuário e cria uma sessão.
Auth: nenhuma (público)
Body: { "username": "string", "password": "string" }
Response 200: { "id": "uuid", "username": "string", "role": "master|user|webservice", "representanteId": "string|null", "nomeCompleto": "string|null", "companiesIds": ["string"] }
Response 401: credenciais inválidas

### POST /api/logout
Encerra a sessão atual.
Auth: sessão
Response 200: vazio

### GET /api/user
Retorna o usuário autenticado pela sessão atual.
Auth: sessão
Response 200: objeto User (sem password)
Response 401: não autenticado

### GET /api/user-count
Retorna a quantidade total de usuários cadastrados (usado para verificar se o sistema já foi inicializado).
Auth: nenhuma (público)
Response 200: { "count": number }

---

## MÓDULO: AUTENTICAÇÃO MOBILE (JWT)

### POST /api/mobile/login
Autentica um usuário mobile e retorna tokens JWT.
Auth: nenhuma (público)
Body: { "username": "string", "password": "string" }
Response 200: {
  "accessToken": "string (JWT, validade 15min)",
  "refreshToken": "string (JWT, validade 30 dias)",
  "user": { "id": "uuid", "username": "string", "role": "string", "representanteId": "string|null", "nomeCompleto": "string|null", "companiesIds": ["string"] }
}
Response 400: username e password são obrigatórios
Response 401: credenciais inválidas

### POST /api/mobile/refresh
Renova o accessToken usando o refreshToken.
Auth: nenhuma (público) — o refreshToken é a credencial
Body: { "refreshToken": "string" }
Response 200: { "accessToken": "string (JWT, validade 15min)" }
Response 400: refreshToken obrigatório
Response 401: token inválido ou expirado

### GET /api/mobile/me
Retorna o usuário autenticado pelo JWT.
Auth: JWT Bearer token
Response 200: objeto User (sem password)
Response 401: não autenticado

Retorna um pedido completo com seus itens.

Retorna estatísticas de pedidos (somente master).

### POST /api/erp/pedidos
Cria um ou múltiplos pedidos (aceita objeto ou array).
Auth: API Token Bearer
Body: {
  "idExterno": "string (obrigatório, único)",
  "clienteId": "string (obrigatório, deve existir em clientes)",
  "status": "Pendente|Concluído|Expedido|Faturado|Cancelado",
  "cadastradoEm": number (timestamp ms, opcional),
  "descontos": [{ "tipo": "string", "valor": number (centavos) }] (opcional),
  "produtos": [
    {
      "id": "string (idExterno do produto, deve existir)",
      "precoVendido": number (centavos),
      "grade": [{ "nome": "string (deve existir na grade do produto)", "quantidade": number }]
    }
  ]

## MÓDULO: REPRESENTANTE — ASSISTENTE GT (CHATBOT IA)

O Assistente GT é um chatbot com GPT e Function Calling. A comunicação do chat é feita via WebSocket.

### POST /api/rep/chat/sessions
Cria uma nova sessão de chat.
Auth: sessão ou JWT | Role: user
Response 200: { "id": number, "representanteId": "string", "criadoEm": "string", "titulo": "string|null" }

### GET /api/rep/chat/sessions
Lista sessões de chat do representante.
Auth: sessão ou JWT | Role: user
Response 200: Array de sessões ordenadas por data decrescente

### DELETE /api/rep/chat/sessions/:id
Exclui uma sessão de chat e todo seu histórico.
Auth: sessão ou JWT | Role: user
Params: id (número inteiro)
Response 200: { "success": true }

### GET /api/rep/chat/sessions/:id/messages
Retorna o histórico de mensagens de uma sessão.
Auth: sessão ou JWT | Role: user
Params: id (número inteiro)
Response 200: Array de objetos com os campos:
  - id: number — ID da mensagem
  - role: "user" | "assistant"
  - content: string — texto da mensagem (para voz: transcrição do áudio)
  - criadoEm: string (ISO 8601)
  - hasAudio: boolean — true se a mensagem do usuário foi enviada por voz (player de áudio disponível)
  - hasTts: boolean — true se o TTS da resposta já foi gerado e está em cache
  - transcricao: string | null — transcrição do áudio (igual a content, para mensagens de voz)
  - audioMime: string | null — MIME type do áudio gravado (ex: "audio/webm", "audio/mp4")
Nota: os blobs de áudio (audio_data, tts_data) NÃO são retornados aqui para não sobrecarregar. Use os endpoints /audio e /tts para obtê-los.

### GET /api/rep/chat/messages/:id/audio
Serve o áudio gravado pelo usuário em uma mensagem de voz.
Auth: sessão ou JWT | Role: user
Params: id (número inteiro) — ID da mensagem com hasAudio = true
Response 200: binário de áudio (Content-Type: audio/webm, audio/mp4, audio/mpeg etc.)
Response 404: mensagem não encontrada ou não possui áudio
Obs: Cache-Control private, max-age=86400. Usar quando hasAudio = true na mensagem para exibir player de áudio.

### POST /api/rep/chat/messages/:id/tts
Gera (ou retorna do cache) o áudio TTS da resposta do assistente.
Auth: sessão ou JWT | Role: user
Params: id (número inteiro) — ID da mensagem com role = "assistant"
Response 200: binário de áudio (Content-Type: audio/mpeg)
Obs: Na primeira chamada gera e salva no banco (hasTts passa a ser true). Nas chamadas seguintes retorna do cache imediatamente.

### POST /api/rep/chat/ws-token
Gera um token temporário (30 segundos) para abrir a conexão WebSocket.
Auth: sessão ou JWT | Role: user
Response 200: { "token": "string (32 chars)" }

### WebSocket ws://[host]/ws?token=TOKEN&sessionId=ID
Conexão WebSocket para o chat em tempo real.
Query: token (obtido em /api/rep/chat/ws-token), sessionId (número)
Protocolo:
  ENVIAR: { "type": "message", "content": "texto da pergunta" }
  RECEBER: { "type": "chunk", "content": "trecho de texto" } (streaming)
  RECEBER: { "type": "done" } (fim da resposta)

## CÓDIGOS DE ERRO COMUNS

400 Bad Request: dados inválidos ou campos obrigatórios ausentes
  { "error": "mensagem" } ou { "success": false, "error": "mensagem", "campos": { campo: descricao } }

401 Unauthorized: não autenticado ou token expirado
  { "error": "Nao autenticado" }

403 Forbidden: autenticado mas sem permissão
  { "error": "Acesso restrito ao usuario master" }

404 Not Found: recurso não encontrado
  { "success": false, "error": "idExterno '...' nao encontrado" }

409 Conflict: recurso já existe (idExterno duplicado)
  { "success": false, "error": "idExterno '...' ja existe na entidade ..." }

500 Internal Server Error: erro interno do servidor
  { "error": "mensagem técnica" }

## CONVENÇÕES

- Valores monetários: todos em CENTAVOS (integer). Ex: R$19,90 = 1990
- Metas de venda: em PEÇAS (unidades), não em reais
- Timestamps: Unix timestamp em milissegundos (Number)
- Datas ISO: formato "YYYY-MM-DD" ou "YYYY-MM-DDTHH:mm:ss.sssZ"
- idExterno: identificador de negócio (string), único por entidade, definido pelo ERP/integração
- Paginação: page (começa em 1), limit (max 200), resposta inclui total
- Arrays aceitos: POST nos endpoints ERP aceitam objeto único OU array de objetos

## NOTAS PARA DESENVOLVIMENTO MOBILE

1. Use POST /api/mobile/login para autenticar
2. Salve accessToken (localStorage/SecureStorage) e refreshToken (SecureStorage)
3. Envie "Authorization: Bearer <accessToken>" em todas as requisições
4. Intercepte respostas 401: chame POST /api/mobile/refresh e repita a requisição original
5. O WebSocket do Assistente GT requer um token temporário: POST /api/rep/chat/ws-token antes de abrir a conexão WS
6. Para permissões de microfone (TTS/STT): Android necessita RECORD_AUDIO no AndroidManifest.xml; iOS necessita NSMicrophoneUsageDescription no Info.plist
